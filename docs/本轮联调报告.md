# 本轮 Server + Client 联调报告

## 目标

本轮工作的目标是只使用 **Server + Client**，不依赖 Web 页面，通过脚本直接模拟 `/ws/console` 创建任务，驱动真实手机执行复杂任务：

- `打开设置，把熄屏时间调为10分钟`

同时验证以下主链路：

1. Server 正常启动并接受设备连接
2. Client 识别实机并连接 Server
3. 通过脚本模拟 `/ws/console` 创建任务
4. Server 进入多轮 ReAct：reason → act → observe → 下一轮 reason
5. Client 执行动作、回 ACK、上传 observation
6. Server 归档 screenshot / react history / chat history / latest log
7. 可通过 Server 接口拉取这些归档结果

## 本轮完成的主要工作

### 1. 修复 Client 动作坐标兼容问题

在真实联调中，Server 下发的动作并不总是使用 Client 原先假定的字段结构：

- 点击类动作可能使用顶层 `x/y`
- 滑动类动作可能使用顶层 `x1/y1/x2/y2`

而 Client 之前更偏向解析：

- 点击：`element`
- 滑动：`start/end`

这会导致实机执行时出现兼容性问题，例如：

- `Missing swipe coordinates`
- `No element coordinates`

为此，本轮在以下适配器中统一补充了解析兼容：

- `Client/src/adapters/adb_adapter.py`
- `Client/src/adapters/hdc_adapter.py`
- `Client/src/adapters/wda_adapter.py`

新增了对以下输入形式的兼容：

- Tap / Double_Tap / Long_Press：兼容 `element` 和顶层 `x/y`
- Swipe：兼容 `start/end` 和顶层 `x1/y1/x2/y2`

这样可以保证 Server 直接产出的动作 payload 可以被 Client 正常消费。

### 6. 修复“明明已经是 10 分钟却不结束”的问题

在上一轮真实联调中，系统已经能成功走到设置页，并看到了与任务高度相关的状态，但模型在识别到目标值后仍继续乱点、乱滑，没有立即结束任务。

为解决这个问题，本轮在：

- `Server/src/services/react_scheduler.py`

中强化了 `DeviceTask.get_system_prompt()` 的规则，明确要求模型在以下情况下必须立即结束：

- 如果屏幕上的明确文本已经表明目标已完成，必须直接输出：
  - `do(action="finish", message="任务已完成")`
- 对于设置类任务，如果界面明确显示目标值已经等于用户要求值，则不要继续点击、滑动或进入其他页面
- 只有在没有明确完成证据时，才允许继续导航查找

这是针对“目标已达成但模型不收敛”的直接修复。

## 本轮仍观察到的注意点

### 1. 归档文件是按设备累计的，不完全是“单次运行独立视图”

例如：

- `history.json`
- `react_records.jsonl`

当前包含了同一设备历史上的多次尝试记录，因此分析最新一次运行时，需要结合本轮单独保存的 `debug_artifacts/.../20260408_131238/` 目录使用。

### 2. `Error: bad size width` 仍偶发出现

该日志仍能在设备能力检查附近看到，但本轮没有阻断主链路，也不是这次任务的当前主阻塞点。

### 3. finish 行为虽然已得到实机结果支持，但仍建议后续继续加强“直接 finish”的可解释性

本轮结果已经是 `completed`，最终界面也确实处在 `10分钟` 已选中状态；但从过程上看，模型仍然存在多走几步、反复点击同一区域的倾向。说明 finish 规则强化有效，但仍有继续优化空间。
